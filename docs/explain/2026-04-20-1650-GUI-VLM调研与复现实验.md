## 1. 调研与选型思路

GUI VLM（Visual Language Model for GUI）兼顾三类能力：

- **Grounding（定位）**：根据自然语言把目标 UI 元素映射为相对坐标 `(x, y) ∈ [0, 1000]`
- **Visual Reasoning（视觉推理）**：理解布局、内容结构、跨区域信息提取
- **Action Planning（动作规划）**：把意图映射为 `click / type / drag` 等动作序列

当前主流商用可调 API（截至 2026-04）：

| 模型 | 来源 | 亮点 |
|------|------|------|
| `doubao-seed-1-6-vision-250815` | 字节火山方舟 | 支持 thinking，综合强 |
| `doubao-1-5-vision-pro-32k-250115` | 字节火山方舟 | 稳定基线，不支持 thinking |
| `qwen3-vl-flash-2026-01-22` | 阿里云百炼 | 轻量、极快、坐标精度意外地强 |
| `qwen3-vl-plus-2025-12-19` | 阿里云百炼 | Qwen 通用多模态最强版 |
| `qwen3-vl-32b-instruct` | 阿里云百炼 | 32B 开源权重规格，非思考 |
| `qwen3-vl-32b-thinking` | 阿里云百炼 | 32B 强制思考，视觉推理能力强 |
| `qwen3.6-flash` | 阿里云百炼 | 全模态 Flash（3.5-flash 升级版），agentic coding / 多模态识别增强 |
| `qwen3.6-plus` | 阿里云百炼 | 全模态 Plus（3.6 最强版），综合能力最强但最慢，Recall 导向 |
| `gui-plus-2026-02-26` | 阿里云百炼 | 面向 agent 的 computer_use 输出 |

## 2. 模型清单与接入参数

### 2.1 Provider 差异

两个 Provider 都暴露 **OpenAI Chat Completions 兼容协议**，用 `openai` Python SDK 统一调用，仅 `base_url` / `api_key` / thinking 参数风格不同。

```text
ARK (火山方舟)        base_url = https://ark.cn-beijing.volces.com/api/v3
DashScope (阿里百炼)  base_url = https://dashscope.aliyuncs.com/compatible-mode/v1
```

| 维度 | ARK | DashScope |
|------|-----|-----------|
| 环境变量 | `ARK_API_KEY` | `DASHSCOPE_API_KEY` |
| thinking 参数 | `extra_body.thinking = {"type": "enabled" or "disabled", "budget_tokens": N}` | `extra_body.enable_thinking = True/False` |
| 视觉增强 | — | `extra_body.vl_high_resolution_images = True` |

### 2.3 调用参数模板

所有请求共用：


### 2.4 GUI-Plus 专用 system prompt

代码：`@e:\MyCode3\ARKAPITEST\src\gui_plus_prompts.py`

- 声明工具 `computer_use` 的 schema（`left_click / type / answer / ...`）
- **声明屏幕 resolution 为 1000×1000** → 模型输出的 `coordinate:[x,y]` 直接是 0-1000，**无需缩放**
- 输出格式：模型会先输出 `Action: ...` 再包一段 `<tool_call>{...}</tool_call>` JSON

### 2.5 坐标输出格式（按厂商官方推荐，**方案 A**）

> **这是本项目的核心工程设计**。不同厂商对 GUI grounding 有各自推荐的坐标 tag，
> 若强行用统一 prompt（如一律要求 `<point>`）会让部分模型走非原生路径、降低精度。
> 我们按模型 `output_format` 字段在 prompt 中填充对应的"官方格式说明段"。
>
> **避免踩坑**：阿里 Qwen3-VL 曾因 prompt 与官方格式不一致在 POS_002 上精度下降 10 分；
> 字节 1-5-vision-pro 曾因被强行要 `<point>` 把 bbox 塞进 point 壳里导致 POS_004 漂移 4 像素不命中。

| `output_format` | Tag 模板 | 坐标空间 | 适用模型 | 官方文档 |
|------|------|------|------|------|
| `point` | `<point>x y</point>` | 0-1000 相对 | `doubao-seed-1-6-vision` | [火山方舟 grounding](https://www.volcengine.com/docs/82379/1616136) |
| `bbox` | `<bbox>x1 y1 x2 y2</bbox>` | 0-1000 相对 | `doubao-1-5-vision-pro` | 同上 |
| `qwen_point` | `<point>(x, y)</point>` 或 `<box>(x1,y1),(x2,y2)</box>` | 0-1000 相对 | qwen3-vl 全系 | [通义 Qwen3-VL Cookbook](https://developer.aliyun.com/article/1685124) |
| `tool_call` | `<tool_call>{"action":"click","coordinate":[x,y]}</tool_call>` | 1000×1000（由 system prompt 声明） | `gui-plus` | Anthropic computer_use 约定 |

#### 2.5.1 Prompt 模板机制

代码：`@e:\MyCode3\ARKAPITEST\src\coord_formats.py`

`TestCase.prompt` 中放置占位符 **`{COORD_FMT}`** 表达"格式说明段"，运行时由
`client.build_messages()` → `build_prompt_for_model()` 根据模型 `output_format`
填入对应文本。区分**单目标**（POS_001a/001b/004）与**多目标**（POS_002/003，`multi_target=True`）两套模板。

示例：同一 prompt 对不同模型展开后的最后一段：

```text
# doubao-seed-1-6-vision  (point)
请以 <point>x y</point> 格式返回该位置的中心坐标，坐标范围为 0-1000...

# doubao-1-5-vision-pro   (bbox)
请以 <bbox>x1 y1 x2 y2</bbox> 格式返回该位置的边界框，坐标范围为 0-1000...

# qwen3-vl-* 全家          (qwen_point)
请以 <point>(x, y)</point> 格式返回该位置的中心坐标，坐标范围为 0-1000...

# gui-plus                 (tool_call)
请使用 computer_use 工具的 click 动作，coordinate 字段填入 [x, y] 的像素坐标。
```

#### 2.5.2 模型实际输出的各种变体（已全部兼容）

`_extract_predictions()` 已兼容的 **9 种**变体（`@e:\MyCode3\ARKAPITEST\src\test_runner.py:503-599`）：

| 代号 | 样例 | 发生场景 |
|------|------|----------|
| A | `<point>100 200</point>` | 标准（doubao-1-6、qwen3-vl 合规时） |
| B | `<point>10 20 30 40</point>` | 部分模型把 bbox 塞进 point 壳（1-5-pro 退化） |
| C | `<point>(100, 200)</point>` / `<point>[100, 200]</point>` | Qwen3-VL 带括号 |
| D | `<point x1="204" y1="59" />` | qwen3-vl-32b-instruct 自闭合属性 |
| D' | `<points x1="925" y1="97" alt="描述">描述</points>` | **qwen3.6-plus** 复数 tag + 非自闭合 + alt 属性 |
| E | `<bbox>10 20 30 40</bbox>` | doubao-1-5-vision-pro 官方 |
| F | `<box>10 20 30 40</box>` / `<box>(10,20),(30,40)</box>` | Qwen3-VL 官方 bbox |
| G | `<tool_call>{"arguments":{"action":"left_click","coordinate":[x,y]}}</tool_call>` | GUI-Plus |
| **H（兜底）** | `(18, 153)` / `[18, 153]` / `18, 153` — **无任何 tag** | **qwen3.6-flash** thinking 模式偶发裸坐标 |
| 特殊 | `<point>(325, 55)` **无闭合** | qwen3-vl-32b-thinking 思考模式下偶发丢 `</point>` |

**H 的安全保障**：使用 `re.fullmatch` 要求整个 content（strip 后）**完全是**一个纯坐标，
绝不会误匹配含推理文本的输出（如 `"坐标大概是(18, 153)"` 会被忽略）。且仅在前 7 种解析均未命中时才触发。

所有解析后统一返回 `{center, bbox}` 在 **0-1000 空间**，无需缩放。

#### 2.5.3 切换到新模型时的清单

添加新模型时需检查：

1. 确认厂商官方 grounding 推荐格式（看 Cookbook / Model Card）
2. 若是现有 4 种之一：直接在 `@e:\MyCode3\ARKAPITEST\src\config.py` 注册 `output_format`
3. 若是新格式：
   - 在 `@e:\MyCode3\ARKAPITEST\src\coord_formats.py` 的 `SINGLE_TARGET_INSTRUCTIONS` / `MULTI_TARGET_INSTRUCTIONS` 两个字典各加一行
   - 在 `@e:\MyCode3\ARKAPITEST\src\test_runner.py` 的 `_extract_predictions()` 添加对应解析分支
   - 写冒烟测试：`uv run python run_tests.py --ids POS_001a POS_004 --models <新模型>`

## 5. 评估指标定义

代码：`@e:\MyCode3\ARKAPITEST\src\test_runner.py:204-277`

### 5.1 坐标解析

`_extract_predictions()` 兼容 **7 种**输出变体，已在 §2.5.2 详列。
关键原则：

- 主正则 `<point>\s*([^<\n]+?)\s*(?:</point>|(?=<)|$)`
  允许**无闭合标签**（如 qwen3-vl-32b-thinking 偶发输出 `<point>(325, 55)`）
- 从 tag 内容中用 `re.findall(r"-?\d+", ...)` 宽松抽取整数，容忍括号/引号/逗号/空格
- `<bbox>` 和 `<box>` 各走独立正则
- `<point x1=".." />` 自闭合属性形式独立解析
- GUI-Plus 的 `<tool_call>` JSON 独立解析，只接受 `click/move/drag` 等需要坐标的动作

### 5.2 评估指标（坐标类）

| 指标 | 定义 | 计算 |
|------|------|------|
| **point_hit_rate** | 预测点命中率（Precision） | `命中预测数 / 预测总数` |
| **gt_coverage_rate** | GT 覆盖率（Recall） | `被覆盖的 GT 数 / GT 总数` |
| **mean_iou** | 平均 IoU（仅模型返回 bbox 时） | 与最匹配 GT 的 IoU 均值 |

命中规则：预测中心点落在 GT bbox 周围 ±5（`POINT_TOLERANCE=5`）容差内即算命中。IoU 阈值 `IOU_HIT_THRESHOLD=0.3`。

### 5.3 评估指标（名称类）

POS_002/RSN_001/RSN_002 有 `EXPECTED_GROUPS` 期望名称列表，按子字符串匹配计算准确率。

## 6. 复现步骤

### 6.1 快速冒烟（单用例单模型，约 2 秒）

```bash
uv run python run_tests.py --ids POS_001a --models qwen3-vl-flash-2026-01-22
```

```bash
uv run python run_tests.py --ids POS_001a POS_001b POS_002 POS_003 POS_004 --models qwen3-vl-32b-thinking
```

### 6.3 完整矩阵（85 次调用，并行 4，约 5 分钟）

```bash
uv run python run_tests.py --thinking-matrix --parallel 4 --task-timeout 120
```

参数说明：

- `--parallel N`：并发任务数（默认 4）。`qwen3-vl-plus` 在 >4 时偶尔触发 RateLimit，推荐不超过 6
- `--task-timeout N`：单任务 HTTP 超时秒数（默认 120）。超时任务会被标记为 `TIMEOUT` 而不阻塞整体
- `--parallel 1`：退化为串行模式，保留详细逐条日志（适合 debug）

### 6.4 生成汇总表（防乱码）

```bash
uv run python scripts/summarize_results.py
```

脚本自动读取 `output/` 目录中最新 `results_*.json`。

### 6.5 可视化 Ground Truth

```bash
uv run python scripts/visualize_gt.py --test POS_003
# → 在 output/ 生成叠加 bbox 的标注图
```

## 8. 关键结论与已知问题

### 8.1 推荐配置

| 使用场景 | 推荐 |
|---------|------|
| 单点 grounding 追求 <1s | `qwen3-vl-flash` off |
| 多点 grounding 高质量 | `qwen3-vl-flash` off 或 `gui-plus` on |
| 细粒度类型过滤（Recall 优先） | `qwen3-vl-32b-thinking`（慢，但找得全） |
| 端到端 agent 动作决策 | `gui-plus` thinking=on |
| 界面深度分析 | `seed-1-6-vision` off（on 太慢不划算） |

### 8.2 已知问题

1. **`qwen3-vl-plus` thinking 触发 `BalanceError`**：需在百炼控制台确认该版本是否已开通，或尝试 `qwen3-vl-plus-thinking` 变体模型名。
2. **`qwen3-vl-flash` 的 POS_003 格式漂移**：从第 2 行起输出 `<point="18" 470</point>`（漏闭合 `>`），不是 prompt 问题，是小模型自回归稳定性问题。
3. **`qwen3-vl-plus` 在 POS_003 "只推理不看图"**：它知道哪些是群组，但坐标靠纯文字推理。若要让 plus 达到 POS_003 100%，需要在 prompt 中强制它"**先在图上定位，再判断类型**"，而非相反。
4. **`qwen3-vl-32b-thinking` 的 `enable_thinking=False` 无效**：我们已在 `client.py` 中对它不发送该参数。但运行时显示 `实际启用 thinking: False`，该字段含义是"我们的请求态度"而非"模型实际行为"，后续可改进。

### 8.3 后续改进方向

### 7.8.5 `qwen3.6-plus` 关键亮点与取舍

**亮点**：
- 🎯 **POS_003 Recall 100% off+on** — **唯一一个 off 模式就能找全所有 3 个目标群组**的模型，适合"宁愿多报不能漏"的场景
- 🏆 **POS_002 thinking=on 100/100 + 名称 91%** — 与 flash 并列顶尖多目标定位
- ✅ RSN 推理准确率全 91%，稳定

**取舍**：
- ⏱ 平均 **37s**，是 flash 的 **9 倍**，plus 的 thinking=on 思考链极长（POS_003 on 生成 11642 token / 218s）
- 📉 POS_003 Precision 略低（off 50%，on 60%） → 典型 Recall-导向行为：输出超过 GT 数量，多报了 3 个非群组

**独特输出格式**：
- off 模式偶发 `<points x1="925" y1="97" alt="...">...</points>`（复数 + 非自闭合 + alt 属性）
- on 模式输出转为标准 `<point>(x, y)</point>` 格式
- 解析器已兼容两种（格式 D'）
